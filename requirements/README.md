# Reproducible Python environment

The flagship environment is CPython 3.11 on Windows x86-64. The committed
`pylock.windows-amd64-cp311.toml` is generated by pip 25.3 from this exact
platform and is not claimed to be portable to another Python or operating
system. `constraints.windows-amd64-cp311.txt` is a conventional exact-version
snapshot of the same validated environment; the pylock remains the stronger
source-and-hash record.

Create the virtual environment outside OneDrive:

```powershell
py -3.11 -m venv "$env:LOCALAPPDATA\venvs\sentinel-unchained-py311"
$python = "$env:LOCALAPPDATA\venvs\sentinel-unchained-py311\Scripts\python.exe"
$env:PIP_REQUIRE_VIRTUALENV = "true"
& $python -m pip install -r requirements/bootstrap.txt
& $python -m pip install -e ".[dev]"
& $python -m pip check
```

Regenerate the platform lock only from a clean CPython 3.11 x86-64 Windows
environment:

```powershell
& $python -m pip lock ".[dev]" -o pylock.toml
Move-Item .\pylock.toml .\requirements\pylock.windows-amd64-cp311.toml -Force
```

pip 25.3 records the local project relative to the output file. After moving
the generated file into `requirements/`, change only the generated
`[packages.directory] path` from `"."` to `".."`; all resolved package URLs,
VCS commit IDs, and hashes remain untouched. This workaround is necessary
because pip 25.3 errors when asked to emit this local-project lock directly
into a child directory.

Acceptance is not merely successful resolution. Run `pip check`, the complete
test suite, Ruff check and format, a wheel build, a second-venv wheel install,
the typed-tool catalog import, `vol -h`, and one real Windows-memory plugin.
The last item remains a separate real-evidence gate.

Do not generate a claimed Windows or Ubuntu lock from a mixed global Python
environment. A future Ubuntu lock must be resolved and verified on the named
Ubuntu/Python target and committed as a distinct file.
